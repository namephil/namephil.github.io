1、什么是JWT

客户端验证，在出现JWT之前，使用token验证客户端
过程：
服务器端验证用户的用户名和密码，生成token给客户端，然后下次客户端请求数据的时候会携带token,服务端收到之后会与之前保存的token进行验证，验证通过后返回数据
出现JWT之后可以把token存储在客户端
过程：

  • 服务端验证客户端的用户名和密码后给客户端发送一个token
  • 客户端把token存储在localstorage中
  • 再次发起请求的时候，用axios拦截器将token放入自定义header authorization中
  • 服务端收到请求并检查是否有authorization，并拿到token用JWT校验
    JWT相当于把数据转换成JSON对象，这个对象分为三部分：头部、负载、签名
  • 头部，保存JWT的元数据，表明所使用的hash算法
  • 负载，用来存放自己的数据
  • 签名： 确保信息完整性

2、浏览器缓存

浏览器向服务器发起请求，服务器返回Expire, Etag, last-modified, max-age
当Expire或max-age过期，就带上if-none-match和if-modified-since
服务器端验证Etag和last-modified,如果一样就返回304未修改，不一样返回新文件

3、CDN缓存

如果接入了CDN的话，DNS解析权也是会改变的，会把DNS解析权交给CNAME指向的CDN专用DNS服务器

其次就是缓存问题，正常情况下，浏览器在发起请求之前，会先查看本地缓存，如果过期的话再去向服务端发起请求；

如果使用了CDN，浏览器会先查看本地缓存，如果未命中，会向CDN边缘节点发起请求，如果没有过期，直接返回数据，此时完成http请求，如果过期，则CDN还需要向源站发起回源请求，来拉取最新数据。

CDN的缓存策略是因服务商的不同而不同的，但是大多会遵循http标准协议，通过Cache-Control里面的max-age来控制缓存时间

CDN优点：减少了用户访问延时，也减少了源站负载
CDN缺点：当网站数据更新时，而此时缓存数据还没有过期，这时候用户只需强制刷新即可获取最新数据。但是如果使用了CDN的话，用户强制刷新也只是请求到CDN上的数据，此时如果CDN没有同步最新数据的话会导致用户访问异常。一般这个时候程序员需要手动更新调用CDN的刷新缓存接口

4、短轮询和长轮询

短轮询：每隔指定的时间，由浏览器向服务器发起http请求，然后服务端返回最新的数据给客户端
长轮询：

  • long poll
    • long poll指客户端发送请求后，服务端没有数据返回，就将其挂起放入队列处理其他请求，直到有数据返回给客户端，然后客户端再次发起请求，以此轮询。
    • 优点：降低无效的网络传输，保证每次都有数据返回
    • 缺点：无法处理高并发，消耗服务器资源严重，服务端不能主动推送。
  • WebSocket
    • 实现客户端和服务器的双向通信，只需要连接一次，就可以互相传输数据，适合实时通讯、数据实时更新。WebSocket数据传输通过frame形式，一个消息可以分成几个片段传输。
    • 优点：双向通信，没有同源限制，数据量轻（第一次请求需要携带请求头，之后不需要），传输效率高
    • 缺点：受网络限制比较大，兼容性差，维护成本高。

5、dns预解析

正常输入 url 地址之后，会进行 dns域名解析，这个过程大概会花费20~120ms，所以这个时候出现了预解析，可以给当前页使用到的其他域名进行预处理，提前进行预解析，这样当再次访问这些域名的时候就不用 dns 解析了
使用方式：

<meta  http-equiv="x-dns-prefetch-control" content="on">  // chrome 火狐新版等浏览器自动为 on，要关闭可以设置为 off
<link  rel="dns-prefetch" href="//www.zhix.net">
<link  rel="dns-prefetch" href="//api.share.zhix.net">
<link  rel="dns-prefetch" href="//bdimg.share.zhix.net">
